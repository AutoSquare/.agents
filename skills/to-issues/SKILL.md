# To Issues（拆成工单）

用纵向切片（追踪弹）把计划拆成可独立领取的工单。

若无问题跟踪与分拣标签词典，本应已配置 —— 请先运行 **setup-matt-pocock-skills** 技能。

## 流程

### 1. 收集上下文

以会话中既有信息为主。若用户以参数传入工单引用（编号、URL 或路径），从问题跟踪拉取正文与评论全文。

### 2. 探索代码库（可选）

若尚未探索，则探索以理解现状。工单标题与描述须使用项目领域术语，并尊重相关 ADR。

### 3. 起草纵向切片

将计划拆成 **追踪弹**工单：每条工单是薄薄一层 **纵向**切片，自上而下贯穿全部集成层级，而非某一层的 **横向**切片。

切片可标为 `HITL` 或 `AFK`。HITL 需要人类介入（架构拍板、设计评审等）；AFK 可由智能体无人工介入地完成并合并。在可行时优先 AFK。

<vertical-slice-rules>
- 每条切片交付狭窄但 **完整** 的贯通路径（schema、API、UI、tests）
- 完成的切片本身可演示或可独立验证
- 宁要许多薄片，不要少数厚片
</vertical-slice-rules>

### 4. 向用户确认

把拟议拆解以编号列表呈现。对每个切片给出：

- **标题**：简短描述性名称
- **类型**：HITL / AFK
- **被阻塞于**：必须先完成的其它切片（若有）
- **覆盖的用户故事**：对应哪些用户故事（若源材料有）

询问用户：

- 粒度是否合适（过粗 / 过细）？
- 依赖关系是否正确？
- 是否要合并或再拆分？
- HITL/AFK 标记是否妥当？

迭代直到用户认可拆解。

### 5. 发布到问题跟踪

对每个通过的切片新建工单，正文使用下方模板。这些工单视作可供 AFK 智能体取用，除非你另有指示，否则发布时使用正确的分拣标签。

按依赖顺序发布（先有阻塞项），这样在「Blocked by」里可填入真实工单标识。

<issue-template>
## Parent

指向问题跟踪上的父工单（若源本就是既有工单则填；否则省略本节）。

## What to build

对该纵向切片的简练描述：端到端行为，而非逐层实现细节。

避免具体路径或代码片段 —— 易过期。例外：若原型片段在状态机、reducer、schema、类型形状等方面比长篇叙述更精确，可在此内嵌并注明出自原型；只保留承载决策的精华 —— 不是完整 demo。
