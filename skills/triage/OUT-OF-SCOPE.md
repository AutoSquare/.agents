# 范围外知识库（`.out-of-scope/`）

仓库中的 `.out-of-scope/` 用来持久记录被拒的功能诉求，承担两件事：

1. **制度记忆** —— 记下拒绝理由，结案后不致遗忘
2. **去重** —— 新工单若与既往拒绝同属一概念，可 **引用**既往结论而无须重演辩论

## 目录结构

```
.out-of-scope/
├── dark-mode.md
├── plugin-system.md
└── graphql-api.md
```

**按概念** 一文件，而非 **按工单**。多个同类诉求归入同一文件。

## 文件格式

文体应舒缓易读 —— 更像短文设计说明，而非数据库行。善用段落、代码片段与例子，使首次阅读的维护者也能懂。

```markdown
# 暗色模式（示例）

本项目不支持暗色模式或面向用户的主题切换。

## 为何超出范围

渲染管线假定单一配色，定义在 `ThemeConfig`。
支持多主题意味着：

- 包裹整棵组件树的主题上下文
- 每组件的主题感知样式解析
- 用户主题偏好的持久化

这是可观的架构偏移，与本项目专注于内容创作的重心不符。
主题由下游嵌入或二次分发输出的消费方操心。

```ts
// 当前 ThemeConfig 未面向运行时切换设计：
interface ThemeConfig {
  colors: ColorPalette; // 单一调色板，构建期解析
  fonts: FontStack;
}
```

## 既往请求

- #42 — 「增加暗色模式」
- #87 — 「无障碍夜间主题」
- #134 — 「深色主题选项」
```

### 文件命名

用简短、描述性的 kebab-case：`dark-mode.md`、`plugin-system.md`。名称应能让人浏览目录便知被拒概念，未必需要打开文件。

### 撰写理由

理由须有实质 —— 不是「我们不想」，而是 **为何**。好的理由引用：

- 项目范围或理念（「项目专注 X；换肤留给下游」）
- 技术约束（「要实现需 Y，与既有 Z 架构冲突」）
- 战略取舍（「我们选 A 弃 B，因为……」）

理由应耐久，避免只说临时状况（「现在太忙」）—— 那只是延期，并非真正的拒绝。

## 何时检视 `.out-of-scope/`

分拣 **第一步：收集上下文** 时，读取 `.out-of-scope/` 全部文件。评估新工单时：

- 请求是否与既有范围外概念匹配
- 匹配靠 **概念相似度**，不靠关键词抠字 —— 「夜间主题」可匹配 `dark-mode.md`
- 若有匹配，向维护者说明：「这类似 `.out-of-scope/dark-mode.md`，先前因……拒绝。您是否仍维持原判？」

维护者可：

- **确认** —— 把新工单加入既有文件的「Prior requests」，再关闭
- **重审** —— 删除或更新范围外文件，工单走正常分拣
- **认为不同** —— 虽有相关但并非同一诉求，照常分拣

## 何时写入 `.out-of-scope/`

仅当 **enhancement**（而非 bug）以 `wontfix` 被拒。流程：

1. 维护者判定功能超出范围
2. 检索是否已有匹配 `.out-of-scope/` 文件
3. 若有：把新工单追加至「Prior requests」
4. 若无：新建文件 —— 概念名、决策、理由、首条先例请求
5. 工单下评论阐明决策并提到 `.out-of-scope/` 文件
6. 以 `wontfix` 关闭

## 更新或删除范围外文件

维护者改变主意时：

- 删除对应 `.out-of-scope/` 文件
- 技能不必重开旧工单 —— 它们只是历史
- 触发重审的新工单照常走分拣流程
