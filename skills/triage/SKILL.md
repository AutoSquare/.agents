# Triage（分拣）

在项目问题跟踪上按小型分拣角色状态机推进工单。

分拣期间发布到问题跟踪的每条评论或工单**必须**以以下声明开头：

```
> *This was generated by AI during triage.*
```

## 参考文档

- [AGENT-BRIEF.md](AGENT-BRIEF.md) —— 如何撰写耐久的智能体简报
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md) —— `.out-of-scope/` 知识库如何运作

## 角色

两类 **类目**角色：

- `bug` —— 坏了
- `enhancement` —— 新能力或改进

五类 **状态**角色：

- `needs-triage` —— 维护者需评估
- `needs-info` —— 等待报告者补充信息
- `ready-for-agent` —— 规格齐备，可由 AFK 智能体认领
- `ready-for-human` —— 需人类实现
- `wontfix` —— 不予处理

每个已分拣工单应恰好携带 **一个**类目角色 **与** **一个**状态角色。若状态角色彼此矛盾，先标出并向维护者确认，再继续。

上述为规范角色名 —— 问题跟踪里实际标签字符串可能不同；映射本应已提供，否则运行 **setup-matt-pocock-skills** 技能。

状态迁移：未打标签工单通常先入 `needs-triage`；再迁到 `needs-info`、`ready-for-agent`、`ready-for-human` 或 `wontfix`。报告者回复后 `needs-info` 回到 `needs-triage`。维护者可随时改写 —— 若迁移看起来反常，先问再动手。

## 调用

维护者唤起 **triage** 技能 并用自然语言说明意图。解析请求并执行。例如：

- 「列出所有需要我关注的内容」
- 「看看 #42」
- 「把 #42 移到 ready-for-agent」
- 「哪些事情可以给智能体领？」

## 展示待关注事项

查询问题跟踪，按 **最旧优先** 呈现三个桶：

1. **Unlabeled** —— 从未分拣。
2. **`needs-triage`** —— 评估进行中。
3. **`needs-info` 且上次分拣笔记后报告者有活动** —— 需要再评估。

显示数量与每条工单一行摘要；让维护者挑选。

## 分拣特定工单

1. **收集上下文。** 读完工单（正文、评论、标签、报告者、日期）。解析先前分拣笔记，勿重复已解问题。用语境术语探索代码库，尊重相关 ADR。阅读 `.out-of-scope/*.md`，若与当前议题相似的既往拒绝理由则提出。

2. **给出建议。** 告诉维护者对类目 / 状态的推荐及理由，附与工单相关的简短代码侧摘要；等待指示。

3. **复现（仅 bug）。** 质询前先尝试复现：读报告步骤、跟踪相关代码、跑测试或命令。汇报结果 —— 成功复现代码路径、失败复现、或细节不足（强 `needs-info` 信号）。确认复现有助于写出更强智能体简报。

4. **质询（若需要）。** 若工单尚欠澄清，跑一次 **grill-with-docs** 技能。

5. **落实结果：**
   - `ready-for-agent` —— 发表评论形式的智能体简报（见 [AGENT-BRIEF.md](AGENT-BRIEF.md)）。
   - `ready-for-human` —— 结构与智能体简报相同，但说明为何不宜委托（需判断、外部访问、设计拍板、人工测试等）。
   - `needs-info` —— 发帖记录分拣笔记（模板见下）。
   - `wontfix`（bug）—— 礼貌解释后关闭。
   - `wontfix`（enhancement）—— 写入 `.out-of-scope/` ，从评论链到该文件，再关闭（见 [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)）。
   - `needs-triage` —— 应用角色；若有半成品进展可附带评论。

## 快速状态改写

维护者若说「把 #42 移到 ready-for-agent」，信任其意图并直接改角色。确认将执行的动作（角色变更、评论、关闭），再执行；跳过质询。若在未质询情况下移到 `ready-for-agent`，询问是否需要撰写智能体简报。

## Needs-info 模板

```markdown
## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- question 1
- question 2
```

务必把质询中已厘清的内容记在「established so far」，避免白费功夫。要问的问题必须具体可执行，不要写「请多给点信息」之类空话。

## 接续先前会话

若工单上已有分拣笔记，先阅读、检查报告者是否回答未决问题，再继续前给出更新全景。勿重复已解之问。
