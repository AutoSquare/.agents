# Diagnose（诊断）

针对难解 bug 的流程。仅在明确有理由时跳过阶段。

探索代码库时，使用项目领域术语表建立相关模块心智模型，并查阅你所触及区域内的 ADR。

## 第一阶段 —— 搭建反馈闭环

**这就是本技能的精髓。** 其余步骤偏机械。若你能为 bug 建立一个快速、确定、可被智能体跑通的通过与失败信号 —— 你就几乎必然能找到根因 —— 二分、验证假设与埋点都只是消费该信号的手段。若没有信号，盯代码再久也徒劳。

在此处投入不成比例的努力。**要狠。要会想。不轻易放弃。**

### 可按大致顺序尝试的搭建方式

1. **失败测试**：触达 bug 所需的任意接缝 —— 单元、集成、e2e。
2. **Curl / HTTP 脚本**：对本地开发服务发请求。
3. **CLI 调用**：fixture 输入，stdout 与已知良好快照 diff。
4. **无头浏览器脚本**（Playwright / Puppeteer）：驱动 UI，断言 DOM/console/network。
5. **回放捕获轨迹**：保存真实请求 / 载荷 / 事件日志至磁盘；在隔离代码路径回放。
6. **一次性底座**：拉起系统最小子集（单服务、mock 依赖），一次函数调用复现路径。
7. **属性测试 / fuzz**：若是「偶尔错误输出」，跑数千随机输入找失败形态。
8. **二分底座**：若 bug 出现于两已知状态（提交、数据集、版本）之间，自动化「启动状态 X、检查」以配合 `git bisect run`。
9. **差分循环**：同输入在老版本与新版本（或两套配置）间跑并 diff 输出。
10. **人机 bash 脚本**（兜底）：必须人工点击时，用 `scripts/hitl-loop.template.sh` 驱动 _他们_ ，使循环仍有结构；捕获的输出仍反馈给你。

搭对反馈环，等于 bug 已修好九成。

### 迭代反馈环本身

把环当成产品。有了 _某种_ 环以后追问：

- 能否更快？（缓存初始化、跳过无关启动、收窄测试范围。）
- 信号能否更锐利？（断言具体症状，而非「没崩溃」。）
- 能否更确定？（固定时间、固定 RNG seed、隔离文件系统、冻结网络。）

30 秒的脆弱环并不比没有强多少；2 秒的稳定环则是调试利器。

### 非确定性 bug

目标不是立刻得到纯洁复现，而是 **提高复现率**。并行循环触发器 100×、加压、收窄时间窗、注入 sleep。50% flaky 仍可调试；1% 不行 —— 持续抬高率直到可用。

### 当真无法搭建闭环时

明确停下并陈述：你已尝试了什么。请用户提供：(a) 能复现的环境访问；(b) 捕获物（HAR、日志 dump、core、带时间戳录屏）；或 (c) 许可临时在线上埋点。**没有闭环就不要进入假设阶段**。

除非你相信手中闭环可靠，否则不要进入第二阶段。

## 第二阶段 —— 重现

跑闭环，观察 bug 出现。

确认：

- [ ] 闭环产生的是 **用户** 描述的失败形态 —— 不是碰巧附近的另一种失败。错 bug = 错误修复。
- [ ] 失败在多轮运行中稳定复现（或对非确定性 bug 有足够高的速率用于调试）。
- [ ] 已捕获确切症状（错误信息、错误输出、慢 timing），以便后续阶段验证修复是否真的对症。

未完成重现前勿前进。

## 第三阶段 —— 假设

在对任一假设动手验证前先生成 **3–5 个排序假设**。单次假设容易被第一个看似合理的想法拴住。

每个假设都必须 **可证伪**：写出它预言的现象。

> 格式：「若原因是 <X>，则 <改 Y> 会让 bug 消失 / <改 Z> 会让其恶化。」

若写不出预言，那是「感觉」 —— 丢掉或 sharpening。

**在测试前把排序列表给用户看。**他们常有 **领域知识** 能瞬时重排（「我们刚部署了 #3 相关改动」），或已知某条假设已排除。成本低、收益大。若用户 AFK 不必阻塞 —— 按你的排序继续。

## 第四阶段 —— 埋点

每次探测必须映射到第三阶段的具体预言。**每次只改变一个变量。**

工具优先级：

1. 若环境支持，**调试器 / REPL 检视**。一个断点胜过十条 log。
2. **有针对性的 log**，置于能区分假设的边界。
3. 绝不「到处打 log 再 grep」。

**为每条调试 log 打上唯一前缀**，例如 `[DEBUG-a4f2]`。收尾时一次 grep 清干净。无标记的 log 会长存；有标记的应删掉。

**性能分支：** 若是性能回归，log 通常是错的。应建立基准测量（计时底座、`performance.now()`、profiler、查询计划），再二分。**先度量，再修。**

## 第五阶段 —— 修复与回归测试

**在修复前先写回归测试** —— 但仅当存在 **正确接缝** 时。

正确接缝是指测试能以 **调用点真实重现** 的方式覆盖 bug。若仅有接缝太浅（单调用方测试但需要多调用方链触发、单元测无法重现触发链），写在那里的回归测试会带来虚假安全感。

**若不存在正确接缝，这本身就是结论。**记下来：架构阻碍了 bug 被打桩。将此标记给下一阶段。

若接缝正确：

1. 把最小复现转成该接缝的失败测试。
2. 确认失败。
3. 施加修复。
4. 确认通过。
5. 用原始（未最小化）场景重跑第一阶段反馈闭环。

## 第六阶段 —— 清理与事后反思

结束前必须：

- [ ] 原始复现不再出现（重跑第一阶段闭环）
- [ ] 回归测试通过（或已记录接缝缺失）
- [ ] 所有 `[DEBUG-...]` 埋点清除（grep 前缀）
- [ ] 一次性原型删除（或迁至明确标注的 debug 区）
- [ ] 最终成立的假设写入 commit / PR 说明 —— 让下一位 debugger 少走弯路

**然后自问：什么东西本可预防此 bug？**若答案牵涉架构变更（无不爽测试接缝、纠缠的调用方、隐蔽耦合），在修复并入后把工作交给 **improve-codebase-architecture** 技能 并附上细节。**建议放在修复完成之后** —— 此时你拥有的信息多于起步时。
