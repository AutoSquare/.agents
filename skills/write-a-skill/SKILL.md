# 撰写 Skills

## 流程

1. **收集需求** —— 向用户确认：
   - skill 覆盖什么任务 / 领域？
   - 要处理哪些具体用例？
   - 需要可执行脚本还是仅指引？
   - 是否附带参考资料？

2. **起草 skill** —— 产出：
   - 简练指引的 SKILL.md
   - 正文若将超 ~500 行则拆附加参考文件
   - 若有确定性操作可附工具脚本

3. **与用户走查** —— 展示草稿并问：
   - 是否覆盖用例？
   - 缺啥或含糊之处？
   - 某节要更详 / 更略？

## 技能目录结构

```
skill-name/
├── SKILL.md           # 主指引（必选）
├── REFERENCE.md       # 详细文档（如需）
├── EXAMPLES.md        # 示例（如需）
└── scripts/           # 工具脚本（如需）
    └── helper.js
```

## SKILL.md 模板

```md
---
name: skill-name
description: 能力简述。在 [触发条件] 时使用。
---

# Skill Name

## Quick start

[最小可行示例]

## Workflows

[复杂任务逐步流程 + checklist]

## Advanced features

[拆到单独文件：见 [REFERENCE.md](REFERENCE.md)]
```

## description 要求

description **是智能体判断是否加载本 skill 时唯一看到的短文**。它与其它已装 skills 的描述一起出现在系统提示里。

**目标：** 恰好足够让其知道：

1. 技能提供何种能力
2. 何时 / 为何要触发（关键词、上下文、文件类型）

**格式：**

- 最长 1024 字符
- 第三人称
- 第一句：做什么
- 第二句：`在……时使用`（具体触发）

**佳例（中文）：**

```
从 PDF 抽取文本与表格，填写表单并合并文档。在用户处理 PDF、提到表单或文档抽取时使用。
```

**劣例：**

```
Helps with documents.
```

无法与其它文档技能区分。

## 何时加脚本

在以下情形添加工具脚本：

- 操作确定性高（校验、格式化）
- 同一段生成代码会反复复用
- 需要显式错误处理

脚本省 token，相较每次生成代码更可靠。

## 何时拆文件

在以下情形拆出附属文件：

- SKILL.md 将超 ~100 行
- 内容分属不同领域（如财务 schema vs 销售 schema）
- 高级能力很少启用

## 走查清单

起草后自检：

- [ ] description 含触发语（Use when… / 在…时使用）
- [ ] SKILL.md 压在 ~100 行内（或已合理分拆）
- [ ] 无时效性信息（会迅速过期的事实）
- [ ] 术语一致
- [ ] 有具体示例
- [ ] 引用链接仅一层纵深
