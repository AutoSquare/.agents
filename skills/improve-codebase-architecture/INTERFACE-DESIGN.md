# Interface Design（接口设计）

当用户想为已选加深候选探索 **多种接口形态** 时，采用此并行子智能体范式。脱胎于「Design It Twice」（Ousterhout）—— 第一直觉往往并非最佳。

沿用 [LANGUAGE.md](LANGUAGE.md) 词汇：**module**、**interface**、**seam**、**adapter**、**leverage**。

## 流程

### 1. 框定问题空间

在派发子智能体前，为该候选撰写 **面向用户** 的问题空间说明：

- 新接口必须满足的约束
- 将依赖哪些外部物，归属哪一依赖类别（见 [DEEPENING.md](DEEPENING.md)）
- 粗略示意代码速写以钉住约束 —— 不是提案，只为让约束具象

展示给用户，然后立刻进入第二步。用户阅读思考的同时子智能体并行工作。

### 2. 派发子智能体

用 Agent 工具 **并行** 启动 3+ 子智能体。每个都要产出 **显著不同** 的加深模块接口。

给每个子智能体独立技术简报（路径、耦合细节、依赖类别自 [DEEPENING.md](DEEPENING.md)、接缝背后是什么）。简报与第一步用户向说明分立。赋予 **不同设计约束**：

- Agent 1：「最小化接口 —— 入口至多 1–3 个。每入口杠杆最大化。」
- Agent 2：「灵活性优先 —— 覆盖许多用例与扩展。」
- Agent 3：「为最常见调用方优化 —— 默认路径要极简。」
- Agent 4（如适用）：「围绕 ports & adapters 处理跨接缝依赖。」

简报同时载入 [LANGUAGE.md](LANGUAGE.md) 与 CONTEXT.md 词汇，使命名与架构语言、领域语言一致。

每个子智能体输出：

1. Interface（类型、方法、参数 —— 另附不变量、次序、错误形态）
2. Usage example（调用方如何使用）
3. 接缝背后由实现藏着什么
4. 依赖策略与 adapters（见 [DEEPENING.md](DEEPENING.md)）
5. Trade-offs —— 何处杠杆厚、何处薄

### 3. 展示与比对

逐项呈现设计让用户消化，再用文字并排对比。按 **depth**（接口处杠杆）、**locality**（改动集中何处）、**seam placement**（接缝安放）对照。

比对后给出 **你的**推荐：哪款最强及为何。若多款可杂交出彩，提议混合方案。**要有主见** —— 用户要强观点，不是要菜单。
