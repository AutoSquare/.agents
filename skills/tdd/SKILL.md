# Test-Driven Development（测试驱动开发）

## 理念

**核心原则：** 测试应透过 **公开接口** 校验 **行为**，而非实现细节。实现可整段重写；测试不应因此而碎。

**好测试** 偏集成风味：走过真实代码路径、使用公开 API。描述系统 **做什么**，不是 **怎么做**。好测试读起来像规格 —— 「用户可用合法购物车结账」直白交代能力是否存在。重构后能活，因为它们不在意内部结构。

**坏测试** 绑在实现上：mock 内部协作方、测私有方法，或绕过接口查证（例如直查数据库）。警讯： refactor 并未改变对外行为测试却炸了。若重命名内部函数测试就红灯，那是在测实现。

示例见 [tests.md](tests.md)，mock 准则见 [mocking.md](mocking.md)。

## 反模式：横向切片

**不要先写齐全部测试再写齐全部实现。** 这是「horizontal slicing」 —— 把 RED 等同「写所有测试」，GREEN 等同「写所有代码」。

会产出 **垃圾测试：**

- 批量写的测试校验 **想象的行为**而非 **真实行为**
- 最终测的是 **形状**（数据结构、函数签名）而非面向用户的行为
- 对真实改动 **迟钝** —— 行为坏了还可能绿，行为没事却红
- **车灯照不到前路**：在理解实现前就锁死测试结构

**正道：** 纵向切片（追踪弹）。一测 → 一实现 → 循环。每条新测试呼应上一轮学到的。正因刚写完代码，你确切知道何谓重要行为以及如何验证。

```
错误（横向）:
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

正确（纵向）:
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
  ...
```

## 工作流

### 1. 规划

探索代码库时使用项目领域术语，令测试命名与接口词汇一致，并尊重相关 ADR。

写代码前：

- [ ] 与用户确认需要哪些接口变更
- [ ] 与用户确认要测的行为（优先级）
- [ ] 寻找 [deep modules](deep-modules.md) 的机会（小接口深实现）
- [ ] 为 [testability](interface-design.md) 设计接口
- [ ] 列出待测行为（不是实现步骤）
- [ ] 取得用户对计划的认可

问：「公开接口应长什么样？哪些行为最值得测？」

**不可能测一切。**与用户确认最关键的若干行为；把精力放在关键路径与复杂逻辑，而非每个边角。

### 2. Tracer Bullet（追踪弹）

写 **一条**测试确认系统的 **一件事**：

```
RED:   为首个行为写测试 → 失败
GREEN: 写尽量少代码通过 → 成功
```

这是追踪弹 —— 证明端到端路径通畅。

### 3. 增量循环

对其余每条行为：

```
RED:   写下一条测试 → 失败
GREEN: 最小代码通过 → 成功
```

规则：

- 一次只加一条测试
- 仅需通过当前测试的代码即可
- 不要替未来测试预写
- 测试聚焦 **可观察行为**

### 4. Refactor

全部绿灯后，寻找 [refactor 候选](refactoring.md)：

- [ ] 抽取重复
- [ ] 加深模块（把复杂度藏在简单接口后）
- [ ] 在顺手处应用 SOLID
- [ ] 思考新代码如何照射旧代码的问题
- [ ] 每步 refactor 后跑测试

**红灯时不要 refactor。**先到全绿。

## 每轮自检

```
[ ] 测试描述行为而非实现细节
[ ] 测试仅使用公开接口
[ ] 测试能扛内部 refactor
[ ] 代码仅为当前测试所需之最小集合
[ ] 未夹带臆测特性
```
