# ADR 格式

ADR 存放在 `docs/adr/`，顺序编号：`0001-slug.md`、`0002-slug.md`，依此类推。

惰性创建 `docs/adr/` 目录 —— 仅在首份 ADR 需要时出现。

## 模板

```md
# {决策的简短标题}

{一至三句：背景是什么、做了什么决定、为何如此。}
```

就这样。ADR 甚至可以只有一段。**价值在于记录「曾有过决定」以及「为何」——** 不是在凑章节。

## 可选章节

仅当真能增值时才加。多数 ADR 用不上。

- **Status** frontmatter（`proposed | accepted | deprecated | superseded by ADR-NNNN`）— 适于会被重访的决策
- **Considered Options** — 仅当被拒方案值得备忘
- **Consequences** — 仅当下游影响不直观时需点明

## 编号

扫描 `docs/adr/` 取最大编号再加一。

## 何时提议 ADR

须**三项全部为真**：

1. **难回滚** —— 反悔成本显著
2. **缺上下文会惊讶** —— 后来者看代码会想「为何会这样」
3. **确有取舍** —— 真实存在备选且因具体理由择一

若易回滚，跳过 —— 你随时能改回去。若不令人惊讶，无人会好奇缘由。若没有真实备选，只剩「显而易见之选」，本无记录必要。

### 典型案例

- **架构形态。**「我们用 monorepo。」「写模型事件溯源，读模型投影进 Postgres。」
- **语境间集成模式。**「Ordering 与 Billing 用领域事件，而非同步 HTTP。」
- **带来锁定的技术选型。** 数据库、消息总线、身份提供方、部署目标。不是每个库 —— 只收录换起来要一两个季度的那种。
- **边界与范围。** 「客户数据归属 Customer 语境；其它语境只持 ID。」明确说不做什么与说做什么同样宝贵。
- **刻意偏离想当然路径。**「我们手写 SQL 不用 ORM，因为……」凡是合理读者会假设相反情形的，都值得记下，以免下一位「好心纠正」 deliberate 设计。
- **代码看不见的约束。**「合规限制不能用 AWS。」「因合作方 API SLA，必须在 200ms 内响应。」
- **拒绝方案本身不直观。** 若考虑过 GraphQL 又因微妙理由选了 REST —— 记下来，免得半年后又有人提案 GraphQL。
