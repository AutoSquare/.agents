# Logic Prototype（逻辑原型）

微型交互终端应用，让用户手推状态模型。适用于 **业务规则、状态迁移或数据形状** —— 纸面看似合理，真推案例才知不对的那些问题。

## 何时选对形态

- 「我不确定这状态机在 X 再 Y 的边角会怎样。」
- 「这数据模型真能否表达……的情形？」
- 「在写真正实现前想把 API 长相处清楚。」
- 任何想让用户 **按键看状态变动** 的情形。

若是「会长什么样」 —— **错分支**。用 [UI.md](UI.md)。

## 流程

### 1. 写下问题

写代码前先声明：原型什么状态模型、回答什么问题。一段话，写在原型 README 或文件头顶注释。答错问题的逻辑原型纯浪费 —— 把问题说死，稍后无论用户临场还是 AFK 回来都能核对。

### 2. 选语言

用宿主项目的语言栈。若没有明显运行时（例如纯文档仓库），先问。

工具链对齐项目惯例 —— 不要仅为原型新开包管理器或运行时。

### 3. 把逻辑隔离为可搬运模块

真正答题的逻辑 —— 放进小型、纯粹的接口背后，理论上可搬进真实代码。**围绕它的 TUI 是一次性的；逻辑本体不应是一次性的。**

形态取决于问题：

- **Pure reducer** —— `(state, action) => state`。动作离散、状态单值时佳。
- **State machine** —— 显式状态与迁移。要问「此刻哪些动作合法」时佳。
- **一组纯函数** 作用在朴素数据类型。无隐式当前状态 —— 仅变换。
- **类或模块**，带清晰方法面，当真需要长期内部状态时。

选对 **回答问题**的形态，而非 **最容易接 TUI**的形态。保持纯：无 I/O、无终端代码、无用来控制流的 `console.log`。TUI import 逻辑单向调用。

这让原型过完寿命仍有价值 —— 问题澄清后可以把 reducer / 机 / 函数集迁入真实模块；TUI 壳删掉。

### 4. 建最小的能摊开状态的 TUI

做成 **轻量 TUI** —— 每拍清屏（`console.clear()` / `print("\033[2J\033[H")` / 等价物）并重绘整帧。用户应只看到稳定的一屏，而不是无限增长的滚屏。

每帧分两节，次序如下：

1. **当前状态**，pretty-print 且 diff 友好（每字段一行，或格式化 JSON）。字段名或区域标题 **加粗**，时间戳 / ID / 推导值可用 **黯淡**。原生 ANSI 即可 —— `\x1b[1m` 粗、`\x1b[2m` 淡、`\x1b[0m` 复位。除非项目已有样式库否则不必新引。
2. **键盘快捷键**，列在底部：`[a] add user  [d] delete user  [t] tick clock  [q] quit`。键名加粗或说明黯淡，择易读者。

行为：

1. **初始化状态** —— 单一内存对象 / 结构体。启动即渲染首帧。
2. **每次读一记键（或一行）**，派发到改写状态的 handler。
3. 每次动作后 **重绘整帧** —— 不要用追加滚屏替换整帧绘制。
4. **循环直到 quit。**

整帧应能装进一屏。

### 5. 一条命令可跑

把脚本加到项目既有任务编排（`package.json` scripts、`Makefile`、`justfile`、`pyproject.toml`）。用户运行 `pnpm run <prototype-name>` 或等价物 —— 不必背路径。

若宿主无任务runner，就把命令写在原型 README 顶栏。

### 6. 交棒

给用户运行命令。他们自己驾驶；妙趣常在「等等，这不应当可能」或「我以为 X 会不同」—— 那是 **想法里的 bug**，正是原型用处。若要加新动作就加；原型演化。

### 7. 捕获答案

尽职后唯一值得留存的是问题的答案。若用户在旁就问学到了什么；若不在，在旁边留 `NOTES.md` 供填 verdict，再删除原型。

## 反模式

- **不加测试。** 需要测试就不再是原型。
- **不接真实数据库。** 除非问题特指持久化，用内存。
- **不泛化。** 没有「以后要支持 X 怎么办」。一次只答一问。
- **勿把逻辑与 TUI 糊在一起。** reducer / 状态机若引用 `console.log`、prompt 或转义序列，便携性归零。TUI 做薄壳包纯模块。
- **别把 TUI 壳带进生产。** 壳为手駕终端优化；背后是可能留下的逻辑模块。
