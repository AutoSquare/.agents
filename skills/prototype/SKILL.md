# Prototype（原型）

原型是 **回答问题的一次性代码**。问题决定形态。

## 选路径

识别正在回答什么问题 —— 来自用户提示、周边代码，或用户在旁时主动询问：

- **「这段逻辑 / 状态模型靠谱吗？」** → [LOGIC.md](LOGIC.md)。搭建微型交互终端，把纸上难推的状态机在真实案例里推一遍。
- **「这应该长什么样？」** → [UI.md](UI.md)。在同一路由下生成几种 **结构迥异**的 UI，用 URL 查询串 + 底部悬浮栏切换。

两条路径产物截然不同 —— 走错路径整次原型作废。若问题本就含糊且联系不上用户，默认同周边代码更匹配的一支（后端模块 → logic；页面或组件 → UI），并在原型顶部写明假设。

## 两条路径共用规则

1. **自 Day 1 起即一次性，且醒目标记。** 把原型放得靠近真实落点（毗邻将接纳它的模块或页面）以便上下文显式 —— 但命名要让路人一眼看出是原型而非生产。一次性 UI 路线遵守项目既有路由惯例；勿凭空发明顶层结构。
2. **一条命令就跑起来。** 用项目现有任务跑法 —— `pnpm <name>`、`python <path>`、`bun <path>` 等。用户不该费脑记路径。
3. **默认无持久化。** 状态常驻内存。持久化本就是原型要 **检验** 的对象，不该成为依赖。若问题显式牵涉数据库，用 scratch DB 或本地文件名里写清「PROTOTYPE — wipe me」。
4. **跳过打磨。** 无测试；错误处理止于让原型 **可跑**；无抽象层。要义是快快学到东西然后删掉。
5. **摊开状态。** 每次动作（logic）或每次变体切换（UI）后，打印或渲染完整相关状态，让用户看清变了什么。
6. **结束即删除或吸纳。** 问题答完要么删，要么把已验证结论折回真实代码 —— 勿让其在仓库发霉。

## 收尾

原型里唯一值得留存的是 **答案**。把问答一道记入耐久处（commit message、ADR、工单，或原型旁的 `NOTES.md`）。若用户在旁，多半只是一句对齐；若不在，留占位便于其（或你在下一轮）删原型前填入结论。
