# UI Prototype（界面原型）

在 **单一路由**下生成 **数种结构上截然不同的 UI 变体**，靠底部悬浮条切换。用户在浏览器翻转变体，择一（或拼装），其余丢弃。

若问题是逻辑 / 状态而非外观 —— **错分支**。用 [LOGIC.md](LOGIC.md)。

## 何时选对形态

- 「这页该长什么样？」
- 「定稿前想看几种控制台方案。」
- 「给设置屏换布局试试。」
- 任何否则会花一整天在脑中比较三张含糊 mock 的情形。

## 两种子形态 —— **强烈优先子形态 A**

UI 原型在 **顶住应用其它部分** 时最容易评判 —— 真顶栏、真侧栏、真数据、真密度。孤悬的一次性路由是真空：每种变体单看都行。只要能合理挂在既有页面，默认子形态 A。仅当prototype真找不到邻近宿主时才用 B。

### 子形态 A —— 调整既有页面（优先）

路由已存在。变体在同一路由渲染，由 `?variant=` URL 查询串门控。数据拉取、params、鉴权照旧 —— 仅渲染子树切换。此为默认；除非有明确理由不选。

若原型对象是尚无页面但 **自然嵌在某一页里的块**（dashboard 新区块、settings 新卡片、现有流的新步骤） —— **仍是 A**。把变体挂进宿主页。

### 子形态 B —— 新页面（最后手段）

仅当被原型之物 **真的没有**可嵌入页 —— 例如全新顶层面，或无法合理内嵌的流程。

按项目路由惯例创建 **一次性路由** —— 勿发明顶层结构。路径或文件名要带明显原型痕迹（例如含 `prototype`）。仍用 `?variant=` 模式。

在选 B 前自问：真的没有可嵌入页？空路由会掩盖 populated 页面本会暴露的设计问题。

两子形态的悬浮底栏相同。

## 流程

### 1. 声明问题并选 N

默认 **3 个变体**。超过 5 个就不再「迥异」而只是噪音 —— 封顶 5。

在原型位置或文件头用一行记下计划：

> 「设置页三种变体，可经 `?variant=` 切换，挂在既有 `/settings`。」

无论用户是否当场反驳都适用。

### 2. 生成彻底不同的变体

起草每一变体。各自必须遵守：

- 页面目的与其可得数据。
- 项目组件库 / 样式体系（Tailwind、shadcn、MUI、纯 CSS……）。
- 清晰导出组件名，如 `VariantA`、`VariantB`、`VariantC`。

变体须在 **结构上**不同 —— 布局、信息层级、首要 affordance 各异，不只换配色。三套微调的卡片网格不是原型，是壁纸。若两稿过近，重写其一并显式指令「禁止使用卡片网格」。

### 3. 接线

在路由放一个总 switcher：

```tsx
// 伪代码 —— 按项目框架改写
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

子形态 A（既有页）：保持 switcher 之上所有数据获取；每变体只换渲染子树。

子形态 B（新页）：在 `/prototype/<name>` 等一次性路由挂同一 switcher。

### 4. 悬浮切换条

屏幕底部居中固定窄条，含三块：

- **左箭头** —— 上轮变体（循环）。
- **变体标签** —— 显示当前键；若组件导出可读名一并显示，如 `B — Sidebar layout`。
- **右箭头** —— 下轮变体（循环）。

行为：

- 点击箭头改写 URL query（框架路由：`router.replace`、`navigate` 等），保证可分享、可 reload。
- 键盘：`←` `→` 也可循环。`<input>`、`<textarea>`、`[contenteditable]` 聚焦时不要劫持方向键。
- 视觉与页面明显区分（高对比胶囊、浅阴影），让人知道不是被评估的设计本身。
- 生产构建隐藏 —— `process.env.NODE_ENV !== 'production'` 或等价判断，防止误合并把条带给用户。

switcher 单文件共享，两子形态复用。放在项目惯例的共享 UI 目录。

### 5. 交棒

抛出 URL（与 `?variant=` 键）。用户自行翻。**「我要 B 的 header 配 C 的 sidebar」** 通常才是真正想要的设计。

### 6. 记录答案并清理

某一变体胜出后记下哪一个及为何（commit、ADR、工单，或 AFK 时在旁 `NOTES.md`）。然后：

- **子形态 A** —— 删掉败选变体与 switcher；把胜者融入既有页。
- **子形态 B** —— 把胜者升格为真实路由，删一次性路由与 switcher。

勿留变体组件或 switcher 尸骸 —— 腐化快且误导读下一位。

## 反模式

- **只在配色或文案上区别的变体。** 那是微调，不是原型。真变体在结构上争论。
- **变体间共享过多代码。** 共享 `<Header>` 尚可；共享整幅 `<Layout>` 就失去比对意义。各变体应能扔掉整幅布局。
- **接真实写入。** 只读原型可以。若要变更，.stub —— 问题是「长相如何」不是「后端通不通」。
- **把原型代码直接升格生产。** 变体写在原型约束下（无测、极简错误处理）。吸纳时重写。
