# 团队统一开发规范

本文档适用于 **Claude Code、OpenCode、Trea** 等运行时。将本文件置于智能体可读位置（例如项目 `CLAUDE.md`、Trea 的 `.agent/rules/`、或 OpenCode 规则目录），或由 `AGENTS.md` 指向此处。

> **Cursor 用户**：请使用 `.agents/rules/cursor/*.mdc`，经安装脚本复制到 `.cursor/rules/`，不要与本 Markdown 混用两套冲突配置。

---

## 1. 语言与命名规范

1. 界面文本（标题、按钮、提示、标签、错误信息）应优先使用中文。
2. 路径、文件名、输出产物文件名应优先使用英文，避免中文路径引发兼容性问题。
3. 文档名称与文档正文必须使用中文。
4. 文档内容中应避免使用「（英文）」等附注式表达，统一采用自然、完整的中文技术表述。

---

## 2. 文档联动与交付要求

1. 任何代码修改完成后，必须同步更新对应开发文档。
2. 若新增代码功能，必须在开发文档中新增对应章节或条目，明确功能说明、实现位置与使用方式。
3. 若项目目录下不存在开发文档，应先创建 `docs` 目录，并新建中文开发文档后再交付代码。
4. 每次新增代码必须在文档中补充对应内容，不得仅提交代码而缺失文档说明。

---

## 3. 开发前置要求

1. 新增代码前，必须先阅读项目目录下的「开发文档」或 `docs` 目录文档。
2. 实施开发时必须遵循文档中已有的规范、约定和归档要求，重点条款优先级最高。

---

## 4. 注释与技术写作规范

1. 代码注释与文档注释应采用专业、客观、可验证的技术表达。
2. 禁止口语化、宣传化、模板化或明显 AI 痕迹表达。
3. 注释应说明设计意图、边界条件、约束关系或关键实现逻辑，避免无信息量描述。

---

## 5. 代码组织与架构规范

1. 代码组织结构应遵循对应语言与框架的主流工程规范。
2. 新增模块、目录或分层方案时，应与现有工程结构保持一致，避免随意扩展。
3. 若项目采用特定组织规范（如分层、模块化、领域划分等），必须在开发文档中明确说明采用标准与执行方式。

---

## 6. 执行检查清单（提交前必须完成）

- 已确认界面文案优先中文，路径与输出文件名优先英文。
- 已完成代码对应文档更新；新增功能已补充对应文档章节。
- 若缺少开发文档，已创建 `docs` 及中文开发文档。
- 已在开发前阅读并遵循项目既有开发规范。
- 注释与文档表述专业、客观，无口语化与 AI 痕迹。
- 代码组织符合语言/框架工程规范，并在文档中说明相关标准（如适用）。

---

## 7. Skills 自动生成规范

- Skills 目录必须包含全部可执行代码，不得依赖项目外部私有模块或宿主项目中的非标准库。Skills 必须具备可迁移性，即脱离当前项目后可直接复制至其他项目独立运行，不得包含硬编码的项目路径、环境变量或宿主项目专属依赖。
- Skills 必须附带完整文档，文档内容须涵盖：全部代码模块的职责说明、各函数的签名与调用方式、参数说明与返回值定义、Agent 调用本 Skills 的示例与操作说明。文档应使任何 Agent 无需阅读源码即可正确调用 Skills 的全部功能。
- Skills 的代码文件与目录组织必须遵循对应语言的主流工程规范（如 Python 采用 src/layout 或 flat layout、TypeScript 采用 src/ 分层结构等），禁止扁平堆砌或随意命名。目录层级、模块划分、命名风格均须符合该语言社区公认的最佳实践。

---

## 8. C# 系列（含 WPF、WinUI3 等）开发规范

1. 所有开发中编写的 `public` 类、接口、结构体、枚举以及 `public` 方法、属性、事件必须添加完整的 XML 文档注释，以避免编译时产生 XML 注释警告（CS1591）。注释内容应遵循以下规范：
   - 使用标准的 XML 文档注释格式，以 `///` 开头
   - 必须包含 `<summary>` 标签，详细描述成员的功能、用途和工作原理
   - 对于方法，应根据需要添加 `<param>`、`<returns>`、`<exception>` 等标签
   - 注释语言应使用简洁准确的中文，避免模糊或冗余描述

   示例：

   ```csharp
   /// <summary>
   /// 用户认证服务，负责处理用户登录、注销和权限验证功能。
   /// 适用于需要用户身份验证的场景，通过调用后端 API 验证用户凭据并维护会话状态。
   /// </summary>
   public class AuthenticationService
   {
       /// <summary>
       /// 用户登录方法
       /// </summary>
       /// <param name="username">用户名</param>
       /// <param name="password">密码</param>
       /// <returns>登录成功返回 true，否则返回 false</returns>
       /// <exception cref="ArgumentNullException">当 username 或 password 为 null 时抛出</exception>
       public bool Login(string username, string password)
       {
           // 实现代码
       }
   }
   ```

2. 代码格式应保持整洁，移除所有不必要的空行。具体规范包括：
   - 类、方法、属性等成员之间只保留一个空行
   - 代码块内部逻辑分组之间可保留一个空行
   - 避免连续多个空行
   - 避免行首、行尾的空白字符
   - 保持一致的缩进风格

## 9. 与本包 `.agents` 的协作

若项目含有 `.agents/`（Trea 下为 `.agent/`）文献与 MCP 配置包，学术类任务另见包内 `workflows.md` 与 `skills/`；本规范侧重**通用工程交付**，二者同时生效时不互相矛盾。

## 10. Git 提交说明与手动提交规范

### 硬性约束

- **禁止 Agent 执行 `git add` / `git commit` / `git push`**，除非用户在同一条消息里**明确写出**「请帮我 commit」或「执行 git 提交」。
- Git **必须由用户本地手动提交**；Agent 只生成可复制文案，并提示去哪个仓库目录执行。
- 仓库根目录通常为 [`GeoPile/`](GeoPile/)（非工作区根）；生成文案前在**实际 git 根**执行 `git status`、`git diff --stat`（只读，不提交）。

### 何时主动提示「可以提交 Git 了」并生成文案

在以下任一情况，**功能收尾回复末尾**须增加一小节「建议手动提交 Git」，并给出完整 commit 文案（用户未禁止时）：

1. 用户明确表示功能结束（如「可以了」「就这样」「功能做完」）。
2. 一次完整任务闭环：实现 + 文档/Agent 交流 +（如有）调试清理完成。
3. Debug / 临时标记清理完毕（如去掉调试日志、恢复正式路径）。
4. 用户说「写 commit 文案 / 提交说明」。

**不要**在每个小步骤都催提交；**一个功能一批改动对应一次建议提交**。

### 生成前必做（只读）

1. `git status --short`、`git diff --stat`（有暂存则 `git diff --cached --stat`）。
2. 阅读 [`Trea调试数据/Agent交流文档/`](Trea调试数据/Agent交流文档/) 中本次任务的「问题 / 处理 / 踩坑」。
3. 对照 `git log -8 --format=%s` 确认语气与仓库习惯一致。
4. 只写 **diff 里真实存在** 的变更；已废弃方案（如后又改全量）不得写入。

### 文案风格（与本仓库历史一致）

以 [`GeoPile`](GeoPile/) 近期 commit 为准，**不要**用 Conventional Commits（禁止 `fix(rebar):` 这类英文 type 标题）。

#### 风格 A：单行（改动面较小或一句话能说清）

- 以 **「修复了…」**、**「优化了…」**、**「新增了…」** 开头。
- 一条完整中文句；可并列多个短句（见历史：`修复了… 修复了…`）。
- 示例（来自仓库）：
  - `修复了开始计算以后不会锁定不允许再点击其他计算的问题`
  - `修复了双筋矩形截面受压筋剖面与竖向大样不出图的问题，优化了桩体构造图子图排布间距，并完善双筋配筋 UI 说明与校验`

#### 风格 B：问题 / 处理（多模块、配筋/出图/一键计算等联动改动）

```
问题：<现象与用户可见后果，一句或几句>
处理：<做了什么；关键文件/模块可点名；须重算/须重编译等操作写在这里>
```

- 「问题」「处理」作为标签；可写在**一段**内（历史 commit 有单行连写）。
- 不必加 Markdown 标题、不必 bullet、不必英文 scope。

#### 禁止

- `fix:` / `feat(scope):` / 英文标题。
- 未验证 diff 就编造文件列表。
- 把 plan 未落地项写进 message。
- Agent 擅自 `git commit`。
- 提交密钥、`.env`、用户私有 `.gpdx`（除非用户明确要求）。

### 输出格式（给用户复制）

```markdown
## 建议手动提交 Git

仓库目录：`GeoPile/`（请自行 `cd` 后 `git add` / `git commit`）

**提交说明（复制整段作为 commit message）：**

\`\`\`
<按风格 A 或 B 生成的正文>
\`\`\`
```

- **不要**附带 `git commit` 命令除非用户要求；默认只给 message 正文。
- 若 `Trea调试数据/` 等不在 GeoPile 子仓库内，须说明「需在对应仓库另提一次」。

### 与用户全局 commit 规则的关系

用户全局规则中的「Agent 仅在明确要求时 commit」与本规则一致；本规则 **更严格**：默认 **永不 commit**，只提醒 + 写文案。

## 11. Karpathy 指南

<!-- karpathy-guidelines:start -->
行为准则，用于减少 LLM 常见编码错误，源自 [Andrej Karpathy 的观察](https://x.com/karpathy/status/2015883857489522876) 关于 LLM 编码陷阱的总结。

**权衡说明：** 这些准则倾向于谨慎而非速度。对于琐碎的任务，请自行判断。

### 1. 编码前思考

**不要假设。不要隐藏困惑。呈现权衡。**

在实现之前：
- 明确说明你的假设。如果不确定，询问。
- 如果存在多种解释，呈现出来 — 不要默默选择。
- 如果存在更简单的方法，说出来。在适当时提出异议。
- 如果有不清楚的地方，停下来。指出困惑之处。询问。

### 2. 简洁优先

**用最少的代码解决问题。不要过度推测。**

- 不要添加要求之外的功能。
- 不要为一次性代码创建抽象。
- 不要添加未要求的"灵活性"或"可配置性"。
- 不要为不可能发生的场景做错误处理。
- 如果写了 200 行而本可以 50 行，重写它。

问自己：「资深工程师会觉得这过于复杂吗？」如果是，简化。

### 3. 精准修改

**只碰必须碰的。只清理自己造成的混乱。**

编辑现有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没坏的东西。
- 匹配现有风格，即使你更倾向于不同的写法。
- 如果注意到无关的死代码，提一下 — 不要删除它。

当你的改动产生孤儿代码时：
- 删除因你的改动而变得无用的导入/变量/函数。
- 不要删除预先存在的死代码，除非被要求。

检验标准：每一行修改都应该能直接追溯到用户的请求。

### 4. 目标驱动执行

**定义成功标准。循环验证直到达成。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入编写测试，然后让它们通过"
- "修复 bug" → "编写重现 bug 的测试，然后让它通过"
- "重构 X" → "确保重构前后测试都能通过"

对于多步骤任务，说明一个简短的计划：
```
1. [步骤] → 验证: [检查]
2. [步骤] → 验证: [检查]
3. [步骤] → 验证: [检查]
```

强有力的成功标准让 Agent 能够独立循环执行。弱标准（"让它工作"）需要不断澄清。

---

**这些准则生效的标志：** diff 中不必要改动更少、因过度复杂而重写更少、澄清问题在实现前而非犯错后提出。
